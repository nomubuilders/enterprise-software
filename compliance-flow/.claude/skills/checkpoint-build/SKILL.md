# Checkpoint Build Execution

Enforce quality-first, checkpoint-based execution for multi-file implementation tasks.

## Prerequisites

The `poa-builder` skill MUST have run first and produced:
- `references/poa.md` — the executable plan of action
- `references/build_log.md` — initialized empty build log

If either file is missing, STOP and run the `poa-builder` skill first.

## Build Log

Update `references/build_log.md` at every checkpoint.

### Log Entry Format

```markdown
## Checkpoint N: [Component Name]
- **Time**: [timestamp]
- **Goal**: [What this checkpoint accomplishes]
- **Files touched**: [list]
- **Status**: PASS | FAIL | BLOCKED
- **Tests**: [test results or N/A]
- **Drift check**: [Does current work align with original goal? YES/NO + explanation]
- **Next**: [What checkpoint N+1 will do]
```

## Execution Loop

1. **Read POA** — Load `references/poa.md`, identify the next incomplete checkpoint
2. **Read build log** — Load `references/build_log.md`, verify previous checkpoints passed
3. **Write tests first** — TDD: create test file/cases for this checkpoint before implementation
4. **Implement** — Write code for this checkpoint only, following the function signatures in the POA
5. **Run tests** — Execute the tests, capture output
6. **Drift check** — Compare current state against the POA goal statement (see rules below)
7. **Log** — Append checkpoint entry to `references/build_log.md`
8. **Gate** — Only proceed if: tests pass AND drift check = YES
9. **Repeat** from step 1

## Drift Detection Rules

Before proceeding to the next checkpoint:

- Re-read the POA goal statement
- Verify every file touched was listed in the POA
- Verify no unplanned files were created
- Verify function signatures match what was planned
- If drift detected: STOP, document the drift in the build log, re-plan before continuing

## Code Quality Gates

Each checkpoint must satisfy ALL of these:

- **DRY**: No duplicated logic — extract shared functions
- **Functional**: Pure functions where possible, minimize side effects, compose small functions
- **TDD**: Test exists before or alongside implementation, tests pass
- **Error handling**: All API paths return structured errors, never raw exceptions
- **Type safety**: All function parameters and returns have type annotations
